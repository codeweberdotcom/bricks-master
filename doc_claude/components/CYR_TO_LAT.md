# Транслитерация кириллицы (cyr-to-lat)

Автоматическая транслитерация кириллических слагов и имён файлов в латиницу.

**Файл:** `functions/cyr-to-lat.php`

---

## Что делает

- При создании/сохранении поста, страницы, термина — автоматически генерирует латинский слаг из кириллического заголовка
- При загрузке файла с кириллическим именем — транслитерирует имя файла на сервере

---

## Хуки

| Хук | Функция | Приоритет |
|-----|---------|-----------|
| `sanitize_title` | `my_cyr_to_lat_slug()` | 10 |
| `sanitize_file_name` | `my_cyr_to_lat_filename()` | 10 |

---

## Функции

### `my_cyr_to_lat_slug($title, $raw_title, $context)`

Хук `sanitize_title`. Транслитерирует слаг если в нём есть кириллица.

**Логика:**

```
$context === 'query'  →  вернуть $title без изменений
empty($title)         →  вернуть $title без изменений
нет кириллицы         →  вернуть $title без изменений
есть кириллица        →  cyr_to_lat($title)
```

**Почему пропускается контекст `query`:**

WordPress вызывает `sanitize_title(..., 'query')` при поиске записи по старому слагу (`_wp_old_slug`). Если транслитерировать в этом контексте — старые URL перестают находиться и редиректы 301 ломаются. Плагин Cyr-To-Lat v6.6.0 содержит этот же фикс.

**Почему обрабатывается `$raw_title`, а не `$title`:**

К моменту вызова фильтра `sanitize_title` WordPress уже прогнал заголовок через `sanitize_title_with_dashes()`, которая URL-кодирует кириллицу: «Привет» → `%d0%bf%d1%80%d0%b8%d0%b2%d0%b5%d1%82`. В `$title` кириллических символов уже нет — `preg_match` ничего не находит и возвращает URL-закодированную строку. `$raw_title` содержит оригинальный заголовок с кириллицей — его и используем.

---

### `my_cyr_to_lat_filename($filename, $filename_raw)`

Хук `sanitize_file_name`. Транслитерирует имя загружаемого файла если в нём есть кириллица.

```
нет кириллицы  →  вернуть $filename без изменений
есть кириллица →  cyr_to_lat($filename)
```

**Зачем нужен:** без этого хука файлы с кириллическими именами сохраняются на сервере как есть (`документ.pdf` → `/uploads/документ.pdf`). Это вызывает проблемы на серверах с не-UTF-8 локалью и при работе с файлами через FTP/SSH.

---

### `cyr_to_lat($text)`

Транслитератор. Принимает строку, возвращает транслитерированный слаг.

**Шаги обработки:**

```
1. strtr($text, $converter)          — заменить кириллические буквы по таблице
2. mb_strtolower($text, 'UTF-8')     — привести к нижнему регистру
3. preg_replace('[^-a-z0-9_]', '-')  — заменить все остальные символы на дефис
4. preg_replace('-+', '-')           — схлопнуть множественные дефисы
5. trim($text, '-')                  — убрать дефисы по краям
```

**Таблица транслитерации (стандарт ГОСТ 7.79-2000, система Б):**

| Кир | Лат | Кир | Лат | Кир | Лат |
|-----|-----|-----|-----|-----|-----|
| а | a | к | k | х | h |
| б | b | л | l | ц | ts |
| в | v | м | m | ч | ch |
| г | g | н | n | ш | sh |
| д | d | о | o | щ | shch |
| е | e | п | p | ь | _(пусто)_ |
| ё | yo | р | r | ы | y |
| ж | zh | с | s | ъ | _(пусто)_ |
| з | z | т | t | э | e |
| и | i | у | u | ю | yu |
| й | y | ф | f | я | ya |

Верхний регистр транслитерируется в те же значения (затем `mb_strtolower` приводит к нижнему).

**Примеры:**

| Вход | Выход |
|------|-------|
| `Новый товар` | `novyj-tovar` |
| `О компании` | `o-kompanii` |
| `FAQ — часто задаваемые вопросы` | `faq-chasto-zadavaemye-voprosy` |
| `документ.pdf` | `dokument.pdf` |
| `Тест 123` | `test-123` |

## Известные ограничения

**Нет префикса у `cyr_to_lat()`** — имя функции без namespace/префикса. Конфликтует с плагином "Cyr-To-Lat" (тот определяет одноимённую функцию-аксессор). Плагин в проекте не используется и устанавливать не планируется — риск неактуален.

**Нет `function_exists()` guard** — защита от двойного объявления отсутствует. Актуально только если появится плагин с тем же именем функции.

**Только русский язык** — таблица содержит исключительно русские буквы. Белорусский, украинский и другие кириллические алфавиты не поддерживаются.
