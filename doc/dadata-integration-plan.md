# План интеграции DaData в тему Codeweber

## 1. Цель

Подключить [DaData API](https://dadata.ru/api/clean/address/) для:
- **Стандартизации адресов** (clean/address) — разбор строки адреса по полям (регион, город, улица, дом, квартира, индекс).
- Опционально: **подсказки адресов** (suggestions) — автодополнение при вводе (требует вызовов с бэкенда из‑за секретного ключа).

## 2. Что уже есть в теме

- **Redux** — секция «API», поле `dadata` (password) для API-ключа в `redux-framework/sample/sections/codeweber/api.php`.
- Ключ хранится в настройках, но **нигде не используется** в коде.
- Для clean/address у DaData нужны **два ключа**: API Token и Secret (X-Secret). В Redux сейчас одно поле — нужно добавить второе для Secret или уточнить формат хранения.

## 3. Ограничения DaData

- **clean/address** — только для адресов в России.
- Вызов из браузерного JS запрещён (чтобы не светить Secret). Запросы — только с бэкенда (PHP).
- В запросе — один адрес за раз; лимиты: 20 запросов/сек, 60 новых соединений/мин с одного IP.

## 4. Этапы интеграции

### 4.1 Настройки Redux

- Добавить второе поле в секцию API (например, `dadata_secret`) для секретного ключа, либо хранить оба значения в одном поле (не рекомендуется).
- Опционально: чекбокс «Включить стандартизацию адресов DaData» и выбор сценариев (только чекаут, только «Редактировать адрес», везде).

### 4.2 Бэкенд (PHP)

- Создать класс или набор функций в `functions/integrations/dadata/` (или рядом с другими интеграциями):
  - Чтение `Redux::get_option($opt_name, 'dadata')` и секрета.
  - Метод `clean_address($address_string)`:
    - POST `https://cleaner.dadata.ru/api/v1/clean/address` с телом `[ $address_string ]`.
    - Заголовки: `Content-Type: application/json`, `Accept: application/json`, `Authorization: Token ${API_KEY}`, `X-Secret: ${SECRET_KEY}`.
  - Разбор ответа (массив с одним объектом): извлечь `region`, `city`, `street`, `house`, `flat`, `postal_code` и т.д. для подстановки в поля формы.
- REST API или AJAX‑обработчик в WordPress:
  - Endpoint (например, `wp_ajax_dadata_clean_address` и `wp_ajax_nopriv_*` при необходимости) принимает одну строку адреса, вызывает `clean_address()`, возвращает JSON с разобранными полями.
  - Проверка nonce и прав (например, только для авторизованных или только для чекаута/редактирования адреса).

### 4.3 Фронтенд (форма адреса)

- **Сценарий A (после ввода адреса):**  
  Кнопка «Проверить и заполнить адрес» рядом с полем «Адрес»: по клику отправить значение `address_1` на бэкенд, получить разобранный адрес и заполнить поля (страна, регион, город, улица, дом, квартира, индекс и т.д.) в форме редактирования адреса / чекаута.
- **Сценарий B (автозаполнение после потери фокуса):**  
  При `blur` поля «Адрес» опционально вызывать тот же endpoint и подставлять результат в остальные поля (с возможностью отключения в настройках).
- Учесть, что страна в форме WooCommerce — отдельное поле (страна уже выбрана); при необходимости передавать её в DaData или вызывать clean только для RU.

### 4.4 Маппинг полей DaData → WooCommerce

| DaData (ответ clean/address) | Поле WooCommerce (billing_*) |
|------------------------------|------------------------------|
| `region`                     | `state` (если подходит формат) |
| `city` / `settlement`        | `city`                       |
| `street_with_type` + `house` | `address_1` (или оставить как есть) |
| `flat`                       | `address_2` или отдельное поле |
| `postal_code`                | `postcode`                   |
| `country_iso_code`           | для проверки страны          |

Решение: подставлять в существующие поля формы «Редактировать адрес» и при необходимости дублировать логику для чекаута.

### 4.5 Обработка ошибок и лимитов

- При 401/403 не светить ключи в ответе; логировать в PHP; вернуть клиенту сообщение «Сервис проверки адреса временно недоступен».
- При 429 (rate limit) — ответ с просьбой повторить позже; не слать повторные запросы автоматически без паузы.
- При пустом или неверном ответе от DaData не перезатирать уже введённые поля без подтверждения пользователя (или только подставлять пустые поля).

### 4.6 Безопасность

- Secret и API Token хранить только на сервере (Redux/опции), не передавать во фронтенд.
- Endpoint защищать nonce и при необходимости проверкой прав (например, только залогиненные пользователи для «Мой адрес»).
- Санитизация входящей строки адреса перед отправкой в DaData (макс. длина, кодировка UTF-8).

## 5. Файловая структура (рекомендуемая)

```
functions/
  integrations/
    dadata/
      class-codeweber-dadata.php   # основной класс: clean_address(), вызов API
      dadata-ajax.php              # регистрация wp_ajax_*, вызов класса
  woocommerce.php                  # подключение хуков для форм адреса (если нужно)
assets/js/
  dadata-address.js                # вызов AJAX, подстановка полей в форму
```

Подключение: в `functions.php` или в отдельном loader подключать класс и AJAX‑обработчик; скрипт `dadata-address.js` подключать только на страницах с формой адреса (чекаут, редактирование адреса).

## 6. Документация и тестирование

- В `doc/` описать: какие ключи где взять (кабинет DaData), куда вписать в Redux, что делает кнопка/автозаполнение.
- Проверить на тестовом ключе лимиты и поведение при неверном/пустом адресе.
- Учесть стоимость: [стандартизация — 20 коп/запрос](https://dadata.ru/api/clean/address/).

## 7. Порядок внедрения

1. Redux: добавить поле для Secret (если ещё не добавлено).
2. PHP: класс вызова cleaner.dadata.ru и разбор ответа.
3. PHP: AJAX‑endpoint и проверки безопасности.
4. JS: кнопка «Проверить адрес» на форме редактирования адреса + подстановка полей.
5. При необходимости: то же для чекаута и сценарий с blur.
6. Документация и тесты.
