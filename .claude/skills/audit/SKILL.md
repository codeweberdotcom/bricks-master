Выполни полный цикл аудита темы CodeWeber по указанному scope: `$ARGUMENTS`.

Допустимые значения scope:
- `blocks` — Gutenberg-плагин (`wp-content/plugins/codeweber-gutenberg-blocks/`)
- `api` — REST API endpoints (плагин + тема)
- `perf` — производительность (тема + плагин)
- `all` — всё перечисленное

---

## Шаг 1: Анализ

Определи scope и запусти соответствующие скиллы:

| scope | скиллы |
|-------|--------|
| `blocks` | `wp-block-development` |
| `api` | `wp-rest-api` |
| `perf` | `wp-performance` + `wp-performance-review` |
| `all` | все четыре |

Для каждого скилла:
- Прочитай его `SKILL.md` из `.claude/skills/[skill]/SKILL.md`
- Изучи соответствующие файлы проекта
- Собери список проблем

Все найденные проблемы сведи в единый список с полями:
- **ID** (P001, P002…)
- **Приоритет**: 🔴 критично / 🟡 важно / 🔵 рекомендация
- **Файл:строка**
- **Описание**
- **Предлагаемое исправление**
- **Тип**: auto (можно исправить автоматически) / manual (требует решения)

---

## Шаг 2: Safety commit

Перед любыми изменениями зафикси текущее состояние:

```bash
cd wp-content/themes/codeweber  # или корень плагина если scope=blocks
git add -A
git commit -m "audit: snapshot before $ARGUMENTS audit"
```

Запомни hash этого коммита — он будет в итоговом отчёте как точка отката.

## Шаг 3: План

Создай файл `doc_claude/reports/YYYY-MM-DD/PLAN-[scope].md`:

```markdown
# Audit Plan — [scope] — YYYY-MM-DD

## Найдено проблем: N
- 🔴 Критично: X
- 🟡 Важно: Y
- 🔵 Рекомендации: Z

## Snapshot commit: [hash]

## Проблемы

| ID | Приоритет | Файл | Описание | Тип |
|----|-----------|------|----------|-----|
| P001 | 🔴 | path/file.php:42 | ... | auto |
| P002 | 🟡 | path/file.js:15 | ... | manual |
...

## Отложено (manual — требует решения)
- P00X: [описание] — [почему требует решения]
```

Покажи план пользователю и **жди подтверждения** перед выполнением.
Спроси: выполнять все auto-пункты, или выбрать конкретные?

---

## Шаг 4: Выполнение

Выполняй только подтверждённые пункты типа `auto`, по порядку приоритета (🔴 → 🟡 → 🔵):

Для каждого пункта:
1. Прочитай файл
2. Внеси исправление — строго по плану, не трогай лишнее
3. Отметь в PLAN-[scope].md: ✅ выполнено / ⏭️ пропущено (причина)

Пункты типа `manual` — помечай 🔶 и выноси в отчёт.

Если исправление затрагивает SCSS или JS — **не запускай build после каждого**,
сборка будет одна в финализации.

---

## Шаг 5: Финализация

Выполни финальный цикл (аналог `/done`):

**5.1 Сборка**
```bash
cd wp-content/themes/codeweber
npm run build
```
Если упала — остановись, сообщи, не коммить.

**5.2 Проверка безопасности**
Для каждого изменённого PHP-файла:
- Нет `$_POST`/`$_GET` без sanitize
- Нет `echo` без `esc_*`
- Нет debug-кода

**5.3 Проверка i18n**
Новые user-facing строки обёрнуты в `__()`/`_e()` с доменом `codeweber`.

**5.4 Обновление документации**
Определи какие файлы `doc_claude/` нужно обновить по изменённому коду и обнови их.

**5.5 Финальный коммит**
```bash
git add -A
git commit -m "audit($ARGUMENTS): fix [N] issues — [краткое описание]"
```
